    项目一：虚拟 IP

    项目概述

    VPN 代理网关设备在转发用户访问服务器的数据时，会使用 VPN 代理网关的IP作为源 IP 去访问业务服务器，此时对业务服务器来说，所有访问请求的数据包源IP都是 VPN 网关的 IP，一旦发生安全事件，无法通过 IP 审计溯源到具体的用户。虚拟 IP 功能能够将用户和虚拟 IP 绑定，解决了业务服务器前置 VPN 代理网关场景下无法审计溯源的问题。

    功能描述

    • 项目主要使用 golang 开发，基于 gin 框架，同时还涉及部分 c 和 lua 代码

    • 项目主要功能点包括：虚拟 IP 申请，释放，续租；共享虚拟IP，独享虚拟 IP；虚拟释放策略与抢占策略；防火墙与路由规则生效等

    项目亮点

    1. 创造性地提出 tproxy 透明代理方案

    https://www.zhaohuabing.com/learning-linux/docs/tproxy/

    包从proxy发往服务器之前通过iptables规则打上标记

    包从服务器回给proxy时通过路由表引流

    上述方案埋下了一个隐患：

    proxy在调用connect() 函数连接上游之前，通过bind() 函数绑定虚拟IP，这个操作会关联一个proxy的端口用作listen，connect() 连接上游的时候也需要占用一个端口，connect()分配端口会跳过 bind() 操作已经使用的端口，由于bind占用的端口过多，导致connect无端口可用。

    2. 修改 linux 内核源码，解决大流量场景源端口不足的问题

    修改内核setsockopt系统调用，新增 sock 选项 SO_BIND_VIP，使用该配置将虚拟IP传到内核中

    其中调用内核 __inet_bind 接口，该接口支持参数（仅绑定IP不绑定端口）

    将原来connect前的bind 操作，修改为 setsockopt 并传入新增的选项 SO_BIND_VIP 以及虚拟IP的地址结构体

    上述方案优缺点：

    • 有效解决问题，改动小，方案足够轻量级
    • 但是导致虚拟IP功能和内核强耦合，其他内核环境也需要一并修改。

    进一步优化思路：

    • 将虚拟IP功能和内核改动解耦

    3. 改进防火墙规则与路由规则，解决虚拟IP池配置过多导致的设备负载高与防火墙规则加载超时的问题

    项目二：TCP隧道代理桌面云流量

    项目概述

    桌面云客户端流量通过深信服 VPN 代理网关进行代理；用户要求对SSL加密流量进行校验，并将校验相关的信息内置于 SSL 的 Client Hello握手包中；代理网关需要解密 TLS 协议的 Client Hello 握手包并向第三方发起校验请求，代理网关建立与上游的 TCP 隧道并代理桌面云流量。

    功能描述

    • 在 openssl 中校验 Client Hello 包的合法性，从 Client Hello 拓展字段获取桌面云客户端插入的校验相关信息，并将其写入堆内存
    • 在 nginx 中从当前 TLS 连接中取出上述校验相关信息，并封装函数供 golang 业务层调用
    • golang 业务层通过 cgo 调用 nginx 层封装的函数，完成对校验相关信息的获取和校验，并通过 golang socket 网络编程实现代理
    • golang 业务层通过 redis 实现对校验结果的缓存，减少不必要的 http 请求，从而提高程序并发性能

    项目亮点

    • 深入 openssl 与 nginx c 语言源码
    • 通过 golang socket 网络编程实现对桌面云客户端流量的代理
    • 利用 redis 缓存机制提高程序并发性能

    技术栈

    golang, c, gin, redis, nginx
