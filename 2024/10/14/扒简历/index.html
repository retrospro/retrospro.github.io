    项目一：虚拟 IP

    项目概述

    VPN 代理网关设备在转发用户访问服务器的数据时，会使用 VPN 代理网关的IP作为源 IP 去访问业务服务器，此时对业务服务器来说，所有访问请求的数据包源IP都是 VPN 网关的 IP，一旦发生安全事件，无法通过 IP 审计溯源到具体的用户。虚拟 IP 功能能够将用户和虚拟 IP 绑定，解决了业务服务器前置 VPN 代理网关场景下无法审计溯源的问题。

    功能描述

    • 项目主要使用 golang 开发，基于 gin 框架，同时还涉及部分 c 和 lua 代码

    • 项目主要功能点包括：虚拟 IP 申请，释放，续租；共享虚拟IP，独享虚拟 IP；虚拟释放策略与抢占策略；防火墙与路由规则生效等

    项目二：TCP隧道代理桌面云流量

    项目概述

    桌面云客户端流量通过深信服 VPN 代理网关进行代理；用户要求对SSL加密流量进行校验，并将校验相关的信息内置于 SSL 的 Client Hello握手包中；代理网关需要解密 TLS 协议的 Client Hello 握手包并向第三方发起校验请求，代理网关建立与上游的 TCP 隧道并代理桌面云流量。

    功能描述

    • 在 openssl 中校验 Client Hello 包的合法性，从 Client Hello 拓展字段获取桌面云客户端插入的校验相关信息，并将其写入堆内存
    • 在 nginx 中从当前 TLS 连接中取出上述校验相关信息，并封装函数供 golang 业务层调用
    • golang 业务层通过 cgo 调用 nginx 层封装的函数，完成对校验相关信息的获取和校验，并通过 golang socket 网络编程实现代理
    • golang 业务层通过 redis 实现对校验结果的缓存，减少不必要的 http 请求，从而提高程序并发性能

    拓展知识

    1. 给我讲解一下端口复用

    端口复用（Port Reuse）是一种允许多个套接字绑定到相同本地IP地址和端口的技术。在标准情况下，操作系统通常会禁止多个进程或套接字绑定到相同的IP地址和端口，以避免冲突。然而，在一些高并发网络应用场景（例如负载均衡、高性能网络服务器等），允许多个套接字复用相同的端口可以提高资源利用效率并简化设计。

    在Linux中，端口复用通常通过两个套接字选项来实现：

    • SO_REUSEADDR：允许多个套接字绑定到同一个本地地址和端口，即使在上一个绑定套接字还处于TIME_WAIT状态时，新的套接字也可以绑定。
    • SO_REUSEPORT：允许多个套接字监听同一个端口，这样多个进程或线程可以共享一个端口号来接收连接请求或数据包，并且操作系统会根据负载自动将流量分发到不同的套接字。

    1. SO_REUSEADDR 详解

    SO_REUSEADDR 允许在一个进程退出后（例如，关闭套接字）或处于 TIME_WAIT 状态时，另一个进程或套接字可以立即绑定到同一个端口。这主要解决的是由于网络连接关闭后的 TIME_WAIT 状态（通常持续 1-4 分钟）而导致端口被暂时占用的问题。

    应用场景

    • 快速重启服务器：当服务器因为某种原因退出（例如崩溃或重启），端口会进入 TIME_WAIT 状态，通常这个状态会持续一段时间，导致在这段时间内无法再次绑定相同的端口。通过设置 SO_REUSEADDR，服务器可以立即重新启动并绑定到同一端口，而不会受到 TIME_WAIT 限制。

    使用示例

    注意事项

    • SO_REUSEADDR 并不允许多个进程同时绑定到同一个端口，除非它们监听的地址不同（如绑定不同的 IP 地址）。
    • 它主要用于解决套接字处于 TIME_WAIT 状态时的端口占用问题，并且不会影响正常的端口使用规则。

    2. SO_REUSEPORT 详解

    SO_REUSEPORT 是 Linux 3.9 引入的一种更强大的端口复用机制，它允许多个套接字完全同时绑定到同一个IP地址和端口。这意味着多个进程或线程可以监听同一个端口，系统会根据负载均衡算法将传入的请求分发到不同的套接字上。与 SO_REUSEADDR 不同，SO_REUSEPORT 允许多个进程或线程共享同一个端口，同时接收连接。

    应用场景

    • 多线程服务器：可以启动多个线程或进程，每个进程都有自己的套接字监听同一个端口。内核会自动将连接请求负载均衡分发给不同的套接字，减少服务器对单个进程的依赖并提高性能。
    • 负载均衡：服务器可以利用多个线程或进程监听同一端口，通过内核自动进行流量分配，提高整体处理能力。

    使用示例

    在这种情况下，假设有多个进程或线程使用相同的代码来创建和绑定套接字，操作系统会在这些进程/线程之间均匀分配传入连接。

    注意事项

    • SO_REUSEPORT 允许多个进程或线程同时监听同一个端口，操作系统会基于负载均衡策略分配连接请求，这可以提升性能。
    • SO_REUSEPORT 适合用在多线程或多进程服务器中，处理高并发场景下的大量连接请求。

    3. 端口复用的好处

    • 提高服务器性能：通过 SO_REUSEPORT，可以启动多个进程或线程同时监听同一个端口，内核会自动负载均衡，提升服务器处理请求的能力，避免单线程或单进程的瓶颈。

    • 解决 TIME_WAIT 问题：通过 SO_REUSEADDR，服务器可以避免因 TIME_WAIT 状态导致的端口占用问题，使服务器能够快速重启而不被端口绑定问题阻碍。

    • 高可用性：允许多个进程或服务复用相同端口，可以在一定程度上提升系统的可用性，尤其在需要热切换或热重启的场景下。

    4. 端口复用的局限性

    虽然端口复用带来了诸多好处，但它并不是在所有情况下都适用，需要根据具体的场景和需求来判断是否应该使用：

    • 安全性问题：如果不慎使用，可能会导致多个进程不合理地共享同一个端口，从而引发混淆或数据冲突问题。

    • 负载均衡机制：SO_REUSEPORT 的负载均衡机制由内核自动决定，通常是根据处理能力（CPU核心数）来进行简单的均衡。在一些复杂的应用场景中，这种均衡策略可能并不能满足需求，可能需要额外的调度机制。

    总结

    端口复用是一种高级的网络编程技术，通过 SO_REUSEADDR 和 SO_REUSEPORT，可以在某些特殊的场景下提高服务器的性能和灵活性。

    • SO_REUSEADDR 允许在 TIME_WAIT 状态下重新绑定端口，主要用于解决快速重启的问题。
    • SO_REUSEPORT 允许多个进程或线程同时绑定到同一个端口，适用于高并发服务器场景，提升系统的扩展性和处理能力。

    不同的应用场景适合使用不同的端口复用机制，需要根据实际需求进行选择和配置。

    2. 在用户态调用bind函数最终会调用到 __inet_bind 函数吗？

    在用户态调用 bind 函数时，最终并不会直接调用到内核中的 __inet_bind 函数，而是通过多个层次的系统调用机制和网络协议栈处理后，可能间接调用到类似 __inet_bind 的函数。下面是从用户态 bind 到内核态实际处理函数的执行过程，详细说明为什么 __inet_bind 是其中的一个环节，而不是直接调用。

    1. 用户态的 bind 调用

    用户进程调用标准的 bind 函数（通常通过 C 库如 glibc 实现），该函数会通过系统调用接口进入内核。这一步是用户态与内核态之间的转换，借助系统调用机制，最终进入内核执行。

    2. 系统调用的处理

    用户态 bind 调用最终触发的系统调用是 sys_bind。它是内核的系统调用入口，用来处理 bind 请求。在 sys_bind 中，内核会解析用户传递的参数（如套接字、地址和端口）并进行初步检查。其伪代码流程如下：

    在这里，sock->ops->bind 是协议相关的 bind 函数，具体由不同的协议族（如 IPv4, IPv6）实现。也就是说，bind 的行为会因套接字类型和协议不同而有所差异。

    3. 套接字层的 bind 函数

    接下来，sock->ops->bind 会根据套接字的类型调用相应协议的实现。例如，对于 IPv4 套接字，sock->ops->bind 指向的是 inet_bind，这是处理 AF_INET 地址族的套接字绑定的函数：

    4. 协议栈中的 bind 实现

    在 inet_bind 函数中，它最终会调用与协议相关的 sk->sk_prot->bind 函数。对于 IPv4 套接字，这个 bind 函数指向的是 inet_bind 函数，而不是直接调用 __inet_bind。在 inet_bind 中，经过一系列的检查之后，才会进一步调用内部的 __inet_bind 函数来执行更底层的绑定操作。

    5. __inet_bind 函数的执行

    最终，inet_bind 会调用 __inet_bind 来处理实际的端口绑定操作。这个函数是底层的实现，负责完成以下任务：

    • 检查指定端口是否可用（是否已经被其他进程占用）。
    • 处理端口冲突、权限检查（如绑定特权端口需要CAP_NET_BIND_SERVICE权限）。
    • 在成功绑定后，设置套接字的状态和端口号。

    __inet_bind 是 Linux 内核网络协议栈中的一个底层辅助函数，它真正执行了绑定操作，但它并不会直接被用户态程序调用。

    6. 总结流程

    用户态 bind 函数的调用过程大致如下：

    1. 用户调用 bind（用户态）。
    2. 触发系统调用 sys_bind（系统调用入口，进入内核态）。
    3. sys_bind 找到与套接字关联的操作函数，并调用 sock->ops->bind。
    4. 对于 IPv4，sock->ops->bind 通常指向 inet_bind，它负责处理具体的 AF_INET 地址族绑定逻辑。
    5. inet_bind 函数中，最终会调用内核的 __inet_bind 函数，完成实际的绑定操作。

    结论

    因此，在用户态调用 bind 函数时，最终会间接调用到内核中的 __inet_bind 函数，但这个过程是经过多个层次的抽象和协议栈处理的，用户态程序并不会直接调用 __inet_bind。

    3. Linux内核源码中，__inet_bind()指定不绑定端口后真的不占用端口吗？ 还是 connect()的时候不会跳过已经经过__inet_bind的端口呢？ 是不是通过 port reuse 端口重用实现的呢？

    在 Linux 内核中，__inet_bind() 函数用于处理 socket 绑定端口的操作，通常在 bind() 系统调用中被使用。当你没有明确绑定一个特定的端口时（例如将端口号设置为 0），系统会在 __inet_bind() 中分配一个随机的可用端口。

    回答你的问题时，我们需要区分以下几点：

    1. “不绑定端口”是否真的不占用端口？

    如果你明确通过 bind() 将端口号设置为 0，或者不调用 bind() 而直接调用 connect()，系统最终都会为你分配一个未被占用的端口。因此，虽然在你看来是“没有绑定端口”，实际上系统已经自动为这个 socket 分配了一个端口，所以它实际上是占用了一个端口的。

    • 如果你不调用 bind() 而直接调用 connect()，内核会自动为该 socket 分配一个临时的随机端口（通常是高端口号）。

    2. connect() 是否会跳过已经经过 __inet_bind 的端口？

    connect() 调用时，如果之前没有调用 bind()，则系统会为 socket 自动分配一个端口。而在端口分配过程中，系统会检查该端口是否已被使用。已经被其他 socket 通过 bind() 绑定的端口不会被重复分配。

    • 换句话说，系统会跳过那些已经被 bind() 占用的端口。

    3. 是不是通过端口重用机制（SO_REUSEADDR, SO_REUSEPORT）实现的？

    端口重用机制（SO_REUSEADDR 和 SO_REUSEPORT）是用于允许多个 socket 在某些条件下同时绑定到相同的 IP 和端口。但在大多数情况下，特别是当你不设置这些选项时，端口分配和使用依然是独立的。

    • 当你设置 SO_REUSEADDR 或 SO_REUSEPORT 时，多个 socket 可以绑定到相同的端口，但这需要特别的设置，并且有特定的使用场景。一般情况下，内核不会自动启用端口重用，除非明确指定。

    总结：

    1. 即使你没有手动绑定一个端口，系统最终会自动分配一个端口，因此“真的不占用端口”的说法是不成立的。
    2. connect() 调用时，系统会跳过那些已经被绑定的端口，分配一个可用的端口。
    3. 端口重用（SO_REUSEADDR, SO_REUSEPORT）是一个额外的机制，允许在特殊情况下多个 socket 同时使用相同的端口，但这不是默认行为。

    因此，端口的选择和分配都是由系统管理的，重用机制只在明确启用时才生效。

    4. IP_BIDN_ADDRESS_NO_PORT 是如何做到仅绑定地址不绑定端口的？这表示这条socket连接真的不占用端口吗？

    在 Linux 内核中，IP_BIND_ADDRESS_NO_PORT 是一个相对较新的选项，它用于允许 socket 仅绑定到 IP 地址而不绑定到特定的端口。具体来说，它会通过 setsockopt() 调用配置 socket 使其能够只绑定 IP 地址，而端口号的分配可以推迟到后续 connect() 或 sendto() 时。

    解释 IP_BIND_ADDRESS_NO_PORT 是如何工作的：

    1. 仅绑定地址而不绑定端口：

      • 使用 IP_BIND_ADDRESS_NO_PORT 选项时，socket 在调用 bind() 时不会分配一个端口，而是仅绑定到指定的 IP 地址。
      • 这意味着，尽管 socket 绑定了 IP 地址，但在 bind() 时它并不绑定具体的端口。端口号的分配会被推迟到稍后的阶段（例如 connect() 或 sendto() 操作时）。

    2. 推迟端口分配：

      • 使用了 IP_BIND_ADDRESS_NO_PORT 后，端口的分配是在 socket 需要真正发送数据时（例如在 connect() 或 sendto() 调用时）才进行。这种行为在一些应用场景下是有用的，比如减少系统中对端口的早期占用，或者减少 bind() 时需要等待分配端口的开销。

    3. 实际的端口分配：

      • 虽然在 bind() 时并没有绑定端口，但 socket 最终仍然会占用一个端口。当你后续调用 connect() 或 sendto() 时，内核会为这个 socket 自动分配一个端口。这与系统在没有明确调用 bind() 时为 socket 自动分配端口的行为类似。
      • 因此，即使设置了 IP_BIND_ADDRESS_NO_PORT，socket 在第一次实际需要与远程主机通信时（比如 connect()）仍然会占用一个端口。

    关键问题的回答：

    1. “这是否表示这条 socket 连接真的不占用端口？”

      • 不，socket 连接最终会占用一个端口。IP_BIND_ADDRESS_NO_PORT 只是在 bind() 调用时不分配端口，而是推迟到后续的 connect() 或 sendto() 操作时才分配端口。所以，从实际的系统资源占用角度来说，socket 在通信时依然会占用一个端口。

    2. “这是如何做到仅绑定地址不绑定端口的？”

      • IP_BIND_ADDRESS_NO_PORT 的实现是在 bind() 时跳过了端口分配逻辑，允许 socket 只绑定 IP 地址而不分配端口。其核心是推迟端口的分配时机，将端口分配放到需要建立连接或发送数据时进行。这样做可以减少在 socket 创建时的端口资源占用，并允许应用程序稍后再确定端口的分配。

    总结：

    IP_BIND_ADDRESS_NO_PORT 允许在 bind() 时仅绑定 IP 地址而不立即分配端口，但最终 socket 还是会占用一个端口，只不过是在后续的 connect() 或 sendto() 之类的操作时才进行分配。因此，socket 最终还是会占用一个端口，只是推迟了分配的时机。

    5. 当我的客户端主机有两张网卡，每张网卡一个IP，这两个IP可以共用同一个端口向服务端的同一个端口发起连接吗？理论上是可以的，这里有两对socket，在四元组中，这两对socket的源IP是不一样的。那么，我的问题是，客户端是如何做到复用同一个端口的，是端口重用吗？

    在你描述的场景下，客户端主机有两张网卡，每张网卡有一个不同的 IP 地址，同时你希望这两个 IP 复用同一个源端口来向服务端的同一个端口发起连接。理论上这是可以的，因为 TCP 连接是通过四元组（源 IP、源端口、目标 IP、目标端口）唯一标识的。在这种情况下，虽然源端口相同，但由于源 IP 不同，因此形成了两个不同的四元组，所以不会冲突。

    回答你的问题：

    客户端在有多个网卡时，确实可以通过不同的源 IP 来复用同一个源端口。这里的复用不是传统意义上的端口重用（SO_REUSEADDR 或 SO_REUSEPORT），而是基于四元组的唯一性。

    1. 四元组是如何避免冲突的？

    在 TCP 协议中，每个连接通过以下四个元素（四元组）唯一标识：

    • 源 IP 地址
    • 源端口号
    • 目标 IP 地址
    • 目标端口号

    由于这两个连接的源 IP 是不同的，即使源端口相同，四元组也不同，因此可以视为两个独立的连接。

    例如：

    • 第一个连接的四元组：<源IP1, 源端口, 目标IP, 目标端口>
    • 第二个连接的四元组：<源IP2, 源端口, 目标IP, 目标端口>

    在这个情况下，虽然源端口相同，但是由于源 IP 不同，这两个四元组是不同的，所以可以复用源端口来建立两个连接。

    2. 客户端是如何做到复用同一个端口的？

    • 当你在不同的网卡（不同的 IP）上使用相同的源端口发起连接时，内核会检查是否四元组是唯一的。如果源 IP 地址不同，即使使用相同的源端口，系统仍然认为这是两个独立的连接。因此，它允许复用这个源端口。

    • 在这种情况下，你不需要设置 SO_REUSEADDR 或 SO_REUSEPORT，因为这些机制通常用于在同一个 IP 地址上复用同一个端口的场景，而你这里是通过不同的源 IP 来复用端口，这属于正常的 TCP 四元组管理逻辑。

    3. 端口重用（SO_REUSEADDR 和 SO_REUSEPORT）和这个场景的区别

    • SO_REUSEADDR：允许多个 socket 绑定到同一个地址和端口，但这通常用于允许一个新的服务器 socket 在旧 socket 还在 TIME_WAIT 状态时重新绑定。这个选项更多地用于服务端的 socket 绑定，不涉及你的场景。

    • SO_REUSEPORT：允许多个进程或线程绑定同一个 IP 和端口，用于负载均衡或高并发场景。同样，它主要用于服务端。

    你这里的场景是通过不同的源 IP 来复用端口，而不是同一个 IP 地址上的端口重用，因此与 SO_REUSEADDR 或 SO_REUSEPORT 无关。

    4. 底层工作机制

    当客户端主机有多个网卡时，内核会根据路由表和指定的网卡或 IP 选择适合的网卡发起连接。假设你绑定了两个不同的 IP 地址，并且每个连接都使用相同的源端口，内核会在创建新的连接时为每个连接生成唯一的四元组。

    由于四元组是唯一的，因此不会发生冲突，系统允许复用同一个源端口。同时，网络栈能够区分不同的连接，因为它们的源 IP 地址不同。

    总结：

    • 客户端主机在有两张网卡并使用不同的源 IP 时，可以复用同一个源端口向服务端发起连接。
    • 这是通过 TCP 的四元组（源 IP、源端口、目标 IP、目标端口）来区分不同连接的，而不是通过传统意义上的端口重用机制（SO_REUSEADDR 或 SO_REUSEPORT）。
    • 复用端口的实现基于四元组的唯一性，因此源端口相同，但由于源 IP 不同，形成了两个不同的四元组，不会产生冲突。
